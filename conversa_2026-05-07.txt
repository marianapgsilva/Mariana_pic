**Como Fica Na LSTM**

A LSTM nao recebe uma linha plana. Ela recebe uma sequencia.

Por exemplo, o doente `4`, para prever o IMC no D563, ficaria assim:

```text
Timestep 1: genero=0, idade=35, dislipidemia=0, dia=0,   imc=50.5651, dias_ate_alvo=563
Timestep 2: genero=0, idade=35, dislipidemia=0, dia=74,  imc=40.5265, dias_ate_alvo=489
Timestep 3: genero=0, idade=35, dislipidemia=0, dia=202, imc=32.3468, dias_ate_alvo=361
Alvo: imc no dia 563 = 32.8673
```

O `reshape_lstm()` faria essa transformacao:

```python
static_count = 3
step_feature_count = 3  # dia, imc, delta_dias
timesteps = len(marcos_input)

def reshape_lstm(data):
    reshaped = np.zeros((data.shape[0], timesteps, static_count + step_feature_count))

    for t in range(timesteps):
        start = static_count + t * step_feature_count
        end = start + step_feature_count

        reshaped[:, t, :static_count] = data[:, :static_count]
        reshaped[:, t, static_count:] = data[:, start:end]

    return reshaped
```

Resumindo: o modelo deixaria de aprender "IMC aos 3 meses" como se fosse
sempre D90. Passaria a aprender "IMC medido no D75", "IMC medido no D97",
"IMC medido no D202", etc. Isto preserva muito melhor a realidade clinica das
medicoes.

```python
def construir_sequencia(static_values, historico, dia_alvo, dia_alvo_estimado=0):
    historico_valido = [
        m for m in historico if np.isfinite(m["dia"]) and np.isfinite(m["imc"]) and m["dia"] < dia_alvo
    ]
    historico_valido = sorted(historico_valido, key=lambda item: item["dia"])[-MAX_TIMESTEPS:]

    seq = np.zeros((MAX_TIMESTEPS, NUM_SEQ_FEATURES), dtype=float)

    for t, medicao in enumerate(historico_valido):
        seq[t] = [
            static_values["genero_num"],
            static_values["idade"],
            static_values["dislipidemia"],
            medicao["dia"],
            medicao["imc"],
            dia_alvo - medicao["dia"],
            medicao.get("dia_estimado", 0),
            dia_alvo,
            dia_alvo_estimado,
            1.0,
        ]

    return seq
```

Sim, ha uma explicacao bastante plausivel.

Aos **3 meses**, o problema e quase "tabular simples". Para prever `3m`, o
modelo praticamente so tem isto:

```text
genero, idade, dislipidemia, IMC inicial, dia alvo
```

Ou seja, ainda **nao existe uma trajetoria real anterior** do doente. A LSTM e
boa a aprender sequencias, padroes temporais e evolucao ao longo do tempo, mas
aos 3 meses ela quase nao tem sequencia para interpretar. Tem basicamente o
ponto inicial e esta a tentar adivinhar o primeiro salto.

Modelos como **Random Forest** e **LightGBM** sao muito fortes neste tipo de
problema tabular curto:

```text
poucas variaveis
relacoes nao lineares simples
pouca sequencia temporal
```

Por isso conseguem ser ligeiramente melhores aos `3m`.

A partir dos **6m, 12m, 24m, 36m...**, a situacao muda. O modelo comeca a ter
uma historia:

```text
D0 -> IMC inicial
D75 -> IMC real aos 3m
D180 -> previsao/medicao dos 6m
D365 -> previsao/medicao dos 12m
...
```

Aqui a pergunta deixa de ser so:

```text
Qual sera o IMC?
```

e passa a ser mais:

```text
Como e que esta curva esta a evoluir ao longo do tempo?
A perda esta a abrandar?
O doente esta abaixo/acima da trajetoria esperada?
Quanto tempo passou entre medicoes?
```

E precisamente aqui que a **LSTM** tem vantagem, porque foi desenhada para
aprender padroes sequenciais. Ela consegue usar a ordem temporal e a relacao
entre medicoes sucessivas de forma mais natural.

Tambem ha outro ponto: quanto mais longe vais no tempo, maior e a importancia
da **dinamica da curva**. Aos 3 meses, a previsao depende muito do ponto
inicial. Aos 24, 36, 48 ou 60 meses, depende mais da forma da trajetoria:

```text
perda inicial rapida
plateau
eventual reganho
estabilizacao
```

A LSTM captura melhor esse comportamento longitudinal.

Portanto, resumindo:

```text
3m:
problema tabular curto -> Random Forest / LightGBM podem ganhar

12m em diante:
problema temporal/longitudinal -> LSTM ganha

global ate 60m:
a vantagem temporal da LSTM compensa a pequena perda no inicio
```

Isto ate e um bom sinal: sugere que a arquitetura esta a aprender algo
especifico da trajetoria, e nao apenas a fazer uma regressao tabular basica.
